# Workflow

Iteration-based development board. File-based, git-friendly, cross-session.

## Структура

```
.workflow/
  ACTIVE                       # одна строка — id активной итерации (например "001")
  iterations/<id>-<slug>/
    README.md                  # цель итерации, scope, exit criteria
    tasks/T###-<slug>.md       # таски
  archive/                     # завершённые итерации (move целиком)
  templates/
    iteration.md
    task.md
```

## Единицы

### Итерация
Один связанный кусок работы с явной целью и exit criteria. Длительность — на твоё усмотрение, обычно 1-2 недели. Одновременно активна **одна** итерация (записана в `ACTIVE`).

### Таск
Одна единица работы. Имеет:
- `id` — `T001`, `T002`, …
- `assignee` — кто делает (`user` или имя зарегистрированного агента из `.claude/agents/`)
- `status` — `todo` | `in-progress` | `review` | `done`
- `deps` — список id, от которых зависит. Стартовать таск с незакрытыми deps нельзя.
- `estimate` — `S` | `M` | `L`
- **Goal** — одно предложение
- **Context** — ссылки на docs/, код, прошлые таски
- **Acceptance criteria** — измеримые условия "готово"
- **How to verify** — конкретные шаги, которые ты пройдёшь руками после возврата агента
- **Notes** — свободный текст, агент добавляет сюда отчёт

## Треки

Параллельно итерациям существуют **треки** — долгоживущие потоки работы (серии ассетов, аудио-паки, длинные пайплайны), которые не умещаются в одну итерацию. Структура:

```
.workflow/tracks/<slug>/
  README.md             # цель, scope, заметки (без exit criteria)
  tasks/T###-*.md       # таски трека, frontmatter содержит `track: <slug>` вместо `iteration:`
.workflow/archive/tracks/<slug>/   # архив (после /archive-track)
```

Правила:
- Треков может быть несколько активных одновременно (в отличие от итераций).
- T### id уникальны на весь проект (итерация + все треки), нумерация глобальная.
- Зависимости (`deps`) могут пересекать границы — таск трека может зависеть от таска итерации и наоборот.
- Архивирование вручную через `/archive-track <slug>`. Нет авто-завершения по exit criteria.
- В web-канбане треки на отдельной вкладке `Tracks`, итерация — на `Iteration`.

## Slash-команды

В `.claude/commands/`:
- `/iter` — статус активной итерации (список тасков, кто делает, что заблокировано)
- `/dispatch <Tid>` — запустить background-agent на таске `<Tid>` по его `assignee` (ищет во всех источниках)
- `/queue` — разгрести `.workflow/queue/` (триггеры от kanban-доски): для каждого dispatch + delete
- `/verify <Tid>` — показать **How to verify** блок таска
- `/new-task` — создать новый таск в активной итерации
- `/new-iter` — создать новую итерацию из шаблона
- `/tracks` — список всех активных треков с метриками
- `/track <slug>` — статус одного трека
- `/new-track` — создать новый трек
- `/new-track-task <slug>` — создать таск внутри трека
- `/archive-track <slug>` — архивировать трек целиком

## Kanban-доска

Web-канбан с drag-drop живёт в `tools/kanban/`. Запуск: `node tools/kanban/server.mjs`, открыть `http://localhost:7777`.

Кнопка `▶ Start` на карточке кладёт триггер в `.workflow/queue/`. В активной Claude-сессии вызови `/queue` — она разгребёт всё. SessionStart hook напоминает о непустой очереди при старте сессии. Подробнее — `tools/kanban/README.md`.

## Ритм рабочей сессии

1. **Старт**: `/iter` → видишь активную итерацию, статусы.
2. **Запуск**: для готовых-к-старту agent-тасков (deps закрыты, status=todo) — `/dispatch T00X`. Можно несколько параллельно.
3. **Свои таски**: пока агенты работают — берёшь свой таск, меняешь его `status: in-progress` руками или через chat.
4. **Возврат агента**: уведомление о завершении → читаешь `Notes` (агент туда отчитался) → `/verify T00X` показывает шаги проверки → проходишь → ставишь `status: done` или возвращаешь в `todo` с комментарием.
5. **Завершение итерации**: когда все таски `done` и exit criteria покрыты — move папку итерации в `archive/`, очисти `ACTIVE`, начни новую через `/new-iter`.

## Правила

- **Atomic status changes**. Меняй `status` после фактического перехода, не до.
- **How to verify обязателен**. Без него таск нельзя `dispatch`-нуть.
- **One assignee per task**. Если работа разделяется — это два таска с deps.
- **Отчёт агента — в Notes**. Не в сторонних файлах. Чтение таска = полная история.
- **deps это запрет, а не подсказка**. `/dispatch` отказывается стартовать таск с открытыми deps.

## Регистрация assignee

`assignee: user` — делаешь сам.

`assignee: <agent-name>` — должен соответствовать файлу в `.claude/agents/<agent-name>.md`. Базовый набор шаблонов:
- `architect` — дизайн решений, планы
- `developer` — реализация фич, рефакторинг, баг-фиксы
- `pencil-designer` — .pen дизайн через Pencil MCP
- `sound-designer` — SFX, UI-кит
- `game-designer` — обсуждение механик и баланса

Добавить нового агента: создать `.claude/agents/<name>.md` с frontmatter (`name`, `description`, `model`, `color`, `memory`). Глобальные агенты — `~/.workflow/agents/`, перекрывают bundled.
