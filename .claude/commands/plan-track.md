allowed-tools: Read, Bash, AskUserQuestion
argument-hint: "<track-slug>"
---

Спланируй итерации в треке: $ARGUMENTS.

Шаги:
1. Через `curl http://127.0.0.1:7777/api/track/<slug>` получи трек: цель, существующие итерации, активная.
2. Прочитай README трека (`.workflow/tracks/<slug>/README.md`) — поле "Цель" и "Scope" дают контекст.
3. Просмотри existing iterations (их status: planned/active/done/abandoned) — поймёшь что уже сделано / запланировано.
4. Через AskUserQuestion уточни у пользователя:
   - **сколько итераций спланировать** (default 3)
   - **горизонт** (что должно быть на выходе после этих N итераций; 1-2 предложения)
   - **подсказки** (если есть направление куда копать дальше)
5. Сгенерируй план: для каждой будущей итерации придумай:
   - короткий kebab-case slug (например `tooltips-mvp`, `polish-pass`, `audio-hookup`)
   - title
   - цель (1-2 предложения)
   - scope (3-7 bullet points)
   - exit criteria (2-4 measurable conditions)
6. Покажи пользователю план списком (slug + title + цель в одну строку каждая). Через AskUserQuestion: ok/правки/отмена.
7. После подтверждения — для каждой итерации сделай POST:
   ```
   curl -s -X POST http://127.0.0.1:7777/api/track/<slug>/iterations \
     -H 'content-type: application/json' \
     -d '{"slug":"<slug>","title":"<title>","status":"planned","body":"<полный markdown с Цель/Scope/Exit criteria/Заметки>"}'
   ```
8. Сообщи: создано N стаб-итераций, видны в kanban → tab Tracks → этот трек. Активировать вручную через UI или `/activate-iter <track> <id>`.

Принципы планирования:
- Маленькие итерации (1-2 недели работы), не "проект" целиком.
- Каждая выдаёт что-то наблюдаемое: демка, билд, прохождение, готовый набор assets.
- Зависимости между итерациями — линейные; если не линейные, обсуди с пользователем.
- Не выдумывай таски — только цели итераций. Таски заполняются позже когда итерация становится active.
